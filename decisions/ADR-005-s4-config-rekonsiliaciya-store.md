# ADR-005 — S4: конфиг-реконсиляция — третья категория контракта store.py

- **Статус:** Accepted
- **Дата:** 2026-07-16
- **Решили:** CEO (Виталий Генаров), по протоколу совещания `meetings/2026-07-16-S4-rubric-repoint.md` (эскалация агента-разработчика при работе над S4)

## Контекст

Тикет S4 (issue #6, `config.yaml` + `config_manager`) требует при перезагрузке конфига перенаправлять `EdgeDef.rubric_id` на новую версию рубрики. Совещание зафиксировало трёхсторонний конфликт артефактов:

1. **ARCHITECTURE §3.5 (до правки):** ровно 4 операции `update_*`, «других не появляется» — repoint запрещён.
2. **data-model §3.2:** «EdgeDef.rubric_id указывает на действующую версию» — repoint предполагается. Дефект класса «самопротиворечивый контракт».
3. **AC тикета S4:** «создаёт/обновляет записи Lesson, ArtifactDef, EdgeDef, Rubric в БД append-only» — внутренне противоречив: Lesson/ArtifactDef/EdgeDef — состояние (DM §3.1), append-only к ним неприменим; «обновляет» требует мутаций шире одного `rubric_id`.

Инвариант И5 (append-only) не затронут: EdgeDef/Lesson/ArtifactDef в И5 не входят — спор шёл о букве контракта §3.5, а не о журнале.

## Рассмотренные варианты

- **А — пятая операция `update_edge_rubric` (узкая заплатка).** Отвергнут: не закрывает тикет — обновления Lesson/ArtifactDef при reload остаются вне контракта, та же развилка эскалировалась бы повторно; «ровно 4» превращается в «сколько понадобится» без критерия. После спора за А не остался никто.
- **Б — действующая рубрика вычисляется: последняя версия по `rubric_key`.** Отвергнут: (а) миграция схемы и новый ключ `rubric_key` не требуются ни одним FR v1 — вето финдира (В8 data-model остаётся открытым); (б) связь ребро↔рубрика становится неявной — «действующая рубрика» как результат сортировки трудно объяснить студенту, оспаривающему вердикт (предметный эксперт); (в) новый класс багов упорядочивания версий («1.10» < «1.9» лексикографически); (г) главное — цель «не трогать контракт» не достигается: Lesson/ArtifactDef всё равно мутируют при reload, §3.5 переписывать в любом случае. **Честный плюс Б** (позиция тестировщика, в решении не учтена): инвариант «нельзя проверить по устаревшей рубрике» держался бы конструкцией, а не тестом.
- **В — контракт §3.5 переписать на три категории** — рекомендация совещания, **принят CEO**.

## Решение

Контракт store.py (ARCHITECTURE §3.5) переписан на **три категории операций**:

1. **Журнал** — только `register_*` (append-only: Rubric, ArtifactSnapshot, CoherenceVerdict, SyncRunRepository, создание Override) — как было.
2. **Рабочее состояние** — ровно 4 узких `update_*` — как было.
3. **Конфиг-реконсиляция из config.yaml** — мутации `Lesson`, `ArtifactDef`, `EdgeDef.rubric_id`; **единственный вызывающий — `config_manager`** (ограничитель подтверждён CEO, закрепляется тестом на импорт).

Порядок работ — документ раньше кода (вето архитектора): сначала ARCHITECTURE §3.5 и AC тикета, потом store.py. Rubric в категорию 3 не входит: правка текста рубрики = новая строка (`register_rubric`).

## Последствия

- **Плюсы:** схема БД не меняется, миграции нет; AC S4 становится проверяемым; рассинхрон §3.5 ↔ data-model §3.2 устранён документами; явная связь ребро↔рубрика — объяснимость вердикта на защите сохранена (вердикт хранит `rubric_id` в четвёрке И3, старые вердикты не пересчитываются и не теряются).
- **Минусы / цена:** гарантия «проверка идёт по действующей рубрике» держится тестом, а не конструкцией — зафиксированное несогласие тестировщика (уверенность 5/10); переписан §3.5 + ссылки в §2/§3.1/§3.2/§9, data-model §3.2, `roles/архитектор.md`; тесты S4: repoint + «старые вердикты нетронуты» + тест на импорт + golden set при смене рубрики.
- **Операционная цена:** смена версии рубрики инвалидирует четвёрки всех пар ребра — до ~25 LLM-пересчётов на ребро следующим сводом; при правках раз в занятие — в бюджете D9/D38.
- **Обязательства AC S4 (вето тестировщика):** AC переписан до кода — reload идемпотентен; мутации только трёх сущностей; вызывающий только config_manager; смена рубрики триггерит прогон golden set (`evals/golden-set.md`); вердикты хранят `rubric_id`.

## Триггеры пересмотра (события, не даты)

| Решение | Пересмотреть, если… |
|---------|---------------------|
| Конфиг-реконсиляция с единственным вызывающим | Появился второй вызывающий конфиг-мутаций кроме config_manager — риск «лазейки» реализовался, вернуть развилку |
| Отказ от `rubric_key` (В8 открыт) | Понадобилась навигация «все версии рубрики X» или golden set перестал сходиться без линейки версий → вернуться к элементам Б |
| Явная связь вместо вычислимой | Тест «repoint при reload» дважды поймал рассинхрон «ребро смотрит на устаревшую рубрику» → пересмотреть в пользу Б |
