# ADR-001 — Ядро FR-5a: LLM-агент по правилам вместо лексического прокси

- **Статус:** принято (CEO, 2026-07-02)
- **Область:** операционализация ядра продукта — детекта разрывов связности (закрывает открытый вопрос №1 PRD, единственный P0 скептик-ревью от 2026-06-30)

## Контекст

Ревью зафиксировало: «хотя бы одна именованная сущность из A встречается в B» неоперационализируемо — агент не напишет детерминированный тест. Рассматривались лексические правила (R1 — идентификаторы, R2 — маркеры markdown) и агентная проверка. CEO сформулировал целевой сценарий: «агент анализирует доки по правилам, я смотрю только в подсвеченные точки».

**Эксперимент 2026-07-02** на реальных парах артефактов проекта:

| Пара (ожидание) | R1 идентификаторы | R2 маркеры | LLM по правилам |
|---|---|---|---|
| user stories → PRD (нет разрыва) | ✅ 20/20 | ✅ 67%, точки-мусор | ✅ + нашёл дефиниционный дрейф «слепой зоны» US-B3↔FR-6 |
| персона Оксана → user stories (разрыв) | вырождается (0 сущностей) | порог «≥1» пропускает; «≥30%» ловит, точки бессмысленны | ✅ 8/9 потеряно + точный фикс («Оксана потеряна молча, остальные исключены явно») |

Лексика не производит осмысленных «точек для взгляда» — целевой сценарий CEO на ней не реализуется. Объём проверок мал (25 репо × ~8 рёбер ≈ 200 пар/нед, ~40 тыс. токенов на пару) — стоимость LLM пренебрежима.

## Решение

1. **Ядро v1 = LLM-агентная проверка ребра по промпт-рубрике.** Сплит FR-5a (лексика) / FR-5b (LLM v2) упраздняется — семантическая проверка входит в v1 как единый FR-5.
2. **Правило проверки (инвариант рубрики):** из источника A извлекаются значимые сущности; каждая получает статус в приёмнике B — **отражена** (где именно; синонимы засчитываются и называются) / **исключена явно** (out-of-scope с обоснованием) / **потеряна** (молча). Разрыв = ≥1 потерянная значимая сущность.
3. **Структурированный выход (контракт):** вердикт (разрыв/нет) · уверенность (выс/сред/низк) · счётчики (проверено/отражено/исключено/потеряно) · ≤5 подсвеченных точек (сущность, цитата из A ≤15 слов, что искали в B и почему не засчитано) · заметки преподавателю ≤3 строк. Эталон промпта — в `evals/golden-set.md`.
4. **Рубрики — данные, не код:** конфиг FR-2 задаёт для каждого ребра рубрику (текст правила) и её **версию**. Смена рубрики/модели = новая версия проверки.
5. **Воспроизводимость (решение CEO):** вердикт привязан к четвёрке `(sha_A, sha_B, версия_рубрики, модель)`. **Перепроверка — только при изменении** любого элемента четвёрки; между обходами вердикт не «мигает». История вердиктов хранится time-series (согласовано с PRD §13, US-C1).
6. **Тестируемость (замена детерминизма):** golden set эталонных пар с известным вердиктом (`evals/golden-set.md`, стартово П1/П3) + метрика ≤20% ложных разрывов (PRD §4). Прогон golden set обязателен при смене рубрики или модели.
7. **ПД (уточнение R-PD3):** в LLM-провайдер уходит только содержимое **публичных** репозиториев (BR-5) и рубрики. Именной реестр (ФИО↔URL) в LLM не отправляется.

## Последствия

- «+» Реализуется целевой сценарий преподавателя; точки разрыва человекочитаемы, с цитатами.
- «+» BR-2 сохраняется в усиленном виде: агент подсвечивает место и объясняет, вердикт об оценке — за преподавателем; override (FR-10) остаётся.
- «−» Зависимость от API LLM: нужен graceful degradation (недоступен API → матрица работает, проверки помечены «отложено»); выбор модели — на архитектурной сессии (вопрос №2).
- Влияние на схему данных (занятие 6): сущность «НаходкаРазрыва» получает поля `уверенность`, `версия_рубрики`, `модель`, точки с цитатами; вердикт неизменяем при неизменной четвёрке.
