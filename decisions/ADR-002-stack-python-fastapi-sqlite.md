# ADR-002 — Стек: Python + FastAPI + SQLite

- **Статус:** Accepted (частично superseded — см. ARCHITECTURE.md v2 §2)
- **Дата:** 2026-07-07
- **Решили:** CEO (по итогам архитектурной сессии, занятие 7)

## Контекст

Выбор стека — единственный открытый вопрос PRD, сознательно вынесенный за его границы. Архитектурные драйверы из `ARCHITECTURE.md` (D1–D39):

| Сила | Драйверы |
|------|----------|
| **Масштаб ничтожен** | 25 студентов, 1 преподаватель, ~450 доков, ~200 LLM-проверок/нед (D1–D16) |
| **Модель — append-only time-series** | История наблюдений, не состояние; сложные инварианты (И1–И11), 12+ сущностей с UUID и enum-доменами (D18, DM) |
| **Интеграции — три внешних API** | GitLab + GitHub (REST), DeepSeek (HTTP, async), CSV-импорт (D31–D34) |
| **Бюджет — $20/мес всего** | VPS $5–8 + DeepSeek $3–5 + домен $1–2 (D36–D37) |
| **VPS — РФ, 1 vCPU, 1–2 GB RAM** | Timeweb/Beget/FirstVDS/Selectel (D37) |
| **Разработка — только LLM-агенты** | D39: spec-first, минимум конфигов, типизированный код; **предсказуемость генерации — ключевой драйвер** |
| **Безопасность — 152-ФЗ** | Шифрование токенов, auth с блокировкой, именных данных в системе нет (D23, D29–D30) |

Главные разноразмерности, которые решает выбор:

1. **D39 vs D36:** стек должен быть максимально предсказуем для LLM-агента (экономия токенов), но укладываться в $5–8 VPS.
2. **D18 vs D37:** append-only с инвариантами (И3 — уникальность четвёрки, И5 — неизменяемость) тянет к реляционной БД с constraints, но 1 vCPU и 1GB RAM не терпят тяжёлой СУБД.
3. **D33 vs D10:** асинхронные LLM-вызовы (~200/нед) не должны требовать отдельной очереди/брокера для одного пользователя.

## Рассмотренные варианты

### A. Python + FastAPI + SQLite (выбран)

| Компонент | Выбор | Обоснование |
|-----------|-------|-------------|
| Язык | Python 3.12+ | Максимальная предсказуемость LLM-агента (D39); зрелые клиенты Git/LLM (D31–D33) |
| Backend-фреймворк | FastAPI | Async из коробки (D33 — LLM-вызовы без Celery); Pydantic — валидация enum-доменов и полей четвёрки (DM §1.10); встроенная документация |
| Хранилище | SQLite (WAL-mode) | Нулевой оверхед (нет отдельного процесса); влезает в 512MB RAM (D37); SQL-constraints покрывают И1–И11; вся БД — один файл — бэкап копированием |
| Фронтенд | Jinja2 + HTMX | Серверный рендеринг — не нужен отдельный API для дашборда; HTMX — обновление матрицы без JS-фреймворка; Alpine.js удалён в ARCHITECTURE v2 (HTMX достаточно для одного пользователя) |
| Аутентификация | `bcrypt` + ручная сессия (FR-0) | Один пользователь — готовое решение (fastapi-users/authlib) тяжелее самой задачи |
| Миграции | Alembic | Единственный стандарт для SQLAlchemy; легковесен |
| Хостинг | VPS (Timeweb/Beget), $5–7/мес | Минимальная конфигурация |

### B. Вариант A1: Python + FastAPI + PostgreSQL

Честные плюсы: настоящие enum-типы, богатые CHECK/partial-индексы для И1–И11 без обходных путей, конкурентная запись без SQLITE_BUSY и рост до 75 студентов (D20) без миграции СУБД.

Отвергнут: PostgreSQL на VPS 512MB не поместится (сам процесс ~200MB); VPS 1GB (+$2–3/мес) — приемлемо, но неоправданно для 450 записей. SQLite в WAL-mode справляется с конкурентностью при правильной архитектуре (один writer, очередь LLM-задач).

### C. Вариант A2: Django + SQLite

Честные плюсы: встроенная админка закрыла бы Override (FR-10) и управление рубриками без ручных форм, а batteries-included auth дал бы FR-0 почти бесплатно.

Отвергнут: Django синхронен — для async LLM-вызовов (D33) потребуется Celery + Redis → +150MB RAM и архитектурный перегруз для 200 вызовов/нед. FastAPI решает async из коробки.

### D. Вариант A3: Django + PostgreSQL

Честные плюсы: самая зрелая и предсказуемая связка для CRUD-приложений в training data (плюс для D39) — админка, ORM и надёжная СУБД в одном стандартном рецепте.

Отвергнут: сочетает минусы B и C — PG + Django + Celery + Redis не влезают в 1GB.

### E. TypeScript + Next.js + SQLite

Честные плюсы: богатейшая интерактивность дашборда (живые обновления матрицы, клиентские фильтры) из коробки React-экосистемы — без ручной сборки HTMX-паттернов.

Отвергнут: Next.js Server Components — оверкилл для одной страницы дашборда; билд требует 2GB RAM (не влезает в 1GB VPS); LLM-агент галлюцинирует границу Server/Client (D39).

### F. Go + templ/HTMX + SQLite

Честные плюсы: один статический бинарник без runtime, минимальное потребление RAM (в разы меньше Python) — идеально под VPS 1 vCPU (D37) с точки зрения эксплуатации.

Отвергнут: экосистема Go для задачи «CRUD + LLM + Git API + dashboard» беднее в training data → агент тратит в 2–3 раза больше токенов (D39 — ключевой драйвер).

## Решение

**Стек v1 (уточнён ARCHITECTURE.md v2):** Python 3.12+ · FastAPI · SQLAlchemy 2.x (только Mapped) + Alembic · SQLite (WAL-mode) · Jinja2 + HTMX · bcrypt + sessions. Alpine.js удалён — HTMX достаточно (D10: один пользователь).

Этот стек:
- **Минимизирует стоимость токенов LLM-агента (D39):** Python/FastAPI/SQLAlchemy — самая широкая представленность в training data; агент реже галлюцинирует API и тратит меньше итераций на типовые паттерны (CRUD, миграции, auth, Git-клиент).
- **Укладывается в $5–7 VPS:** SQLite не требует отдельного процесса; FastAPI + Jinja2 — ~60MB RAM; Alembic — миграции без Docker.
- **Покрывает async LLM-вызовы (D33):** FastAPI BackgroundTasks — очередь без брокера; SQLite WAL-mode выдерживает конкурентное чтение при одном пишущем.
- **Несёт SQLite-риски осознанно:** единственный write-поток (воркер LLM-проверок) + обход репозиториев последовательно; read-запросы дашборда не блокируются благодаря WAL.

## Последствия

### Плюсы
- **Максимальная скорость разработки агентом:** P0-функции (матрица FR-4, аутентификация FR-0, импорт CSV FR-1) — типовые паттерны, агент пишет с первого промпта.
- **Нулевой DevOps:** SQLite — не отдельный сервис; Alembic — `alembic upgrade head`; деплой — `pip install && uvicorn`.
- **Бэкап = `cp` файла:** всё состояние в одном `.sqlite3`.
- **HTMX вместо SPA:** агент пишет HTML-шаблоны, не JSON API — меньше кода, меньше галлюцинаций.

### Минусы / цена
- **SQLITE_BUSY** при конкурентной записи: воркер обхода и воркер LLM-проверок не могут писать одновременно. **Решение:** единая sequential очередь (однопоточный writer). Если в production проявится — перейти на PostgreSQL (A1) ценой +$2–3/мес.
- **Нет готовой админки** (как Django Admin): Override (FR-10), управление рубриками (FR-2) — пишем простые формы.
- **Нет built-in auth:** FR-0 (bcrypt + failed_attempts + locked_until) — ~50 строк кода, но пишем сами.
- **Масштабирование вверх:** при росте до 75+ студентов (D20) SQLite упрётся в конкурентность — потребуется PG.
- **Миграции без типов:** Alembic не проверяет data integrity на уровне типов — golden set (как интеграционный тест) обязателен.

### Что станет сложнее / новый техдолг
- **Ручная сериализация enum-доменов** (artifact_role, verdict_value, snapshot_status… — 11 доменов из DM): SQLite не знает enum. **Решение:** SQLAlchemy `TypeDecorator` + Pydantic — дублирование определения. При смене enum — миграция данных вручную.
- **И3 (уникальность четвёрки):** уникальный composite index на 4 поля — SQLite поддерживает, но без partial indexes для `deferred` (С1 из DM). Открытый вопрос: исключить `deferred` из уникальности или вынести в отдельную таблицу очереди.
- **С1 (deferred — не вердикт):** не решается выбором стека; требует отдельного ADR или доработки схемы.
- **Отсутствие `FOR UPDATE SKIP LOCKED`** в SQLite: при конкурентном чтении очереди LLM-задач — риск двойной обработки. **Решение:** флаг `processing` + `UPDATE ... WHERE processing=0 LIMIT 1` с ретраем.
