# User Stories — преподаватель (Виталий) · цельный сквозной сценарий

> **Вход в занятие 5 (live-сборка PRD).** Артефакт занятия 4: user stories **одного типа пользователя** — главного (преподаватель), оформленные как **единый сценарий**, а не россыпь карточек.
> Источники (трассировка обязательна): `занятие-04-демо/01-персона-и-интервью-виталий.md`, `02-анализ-интервью.md`, `слайд-jtbd.md`, `слайд-стейкхолдеры.md`.
> **Дисциплина:** каждая story тянется назад к JTBD ← боли ← цитате из интервью. Нет трассировки — выкидываем.

## 0. Почему один пользователь и один сценарий

Берём **только преподавателя** (ключевой ЛПР и пользователь). Дамир и Алия — отдельные сценарии, сегодня их не тянем: цель курса — инженерный прототип **одного полного сценария**, а не всё поверхностно. Истории выстроены не списком фич, а как **реальный жизненный цикл инструмента у Виталия**: настроил один раз → каждую неделю смотрит → опирается на это на защите.

**Ядро ценности (сигнал из синтетического интервью — гипотеза, не факт):**
> «Отставание по срокам я прощаю. Разрыв смысла — нет… Найди мне разрывы — читать буду я.»

Значит сердце сценария — предположительно **не матрица "сдал/не сдал"** (это лишь поверхность входа), а **подсветка разрывов связности артефактов**. Матрица — то, с чего взгляд начинается; разрывы — то, ради чего инструмент существует.

> ⚠️ **Все интервью синтетические** (агент играет персону). Это приоритизированные **гипотезы**, а не подтверждённые факты — эхо-камера «агент интервьюирует агента» не заменяет живой custdev. Каждое требование ниже — кандидат на проверку, а не истина.

## 1. Сквозной сценарий нарративом (контекст для всех US ниже)

**Воскресенье, вечер.** Виталий садится готовиться к занятию. Раньше открывал репозитории по одному, успевал ~8–10 из 25, остальные — слепая зона. Сейчас открывает дашборд:

1. **Видит всю группу за минуту** — матрица «репозиторий × занятие», где чего не хватает *(вход взгляда)*.
2. **Сразу видит не галочки, а разрывы смысла** — где у студента требования не бьются со схемой данных, архитектура не из его же плана *(ядро)*.
3. **Ловит слепую зону и хроников** — кого он давно не смотрел; кто отстаёт несколько занятий подряд, хотя в чате тихий.
4. **Доверяет данным**, потому что они читаются из репо сами, без его ручного ввода — иначе «умрёт за три недели».
5. **На защите** студент говорит «я всё сдал вовремя» — Виталий за секунды показывает, *где именно* порвалась цепочка. Доказательная база, а не «придирается».

Этот порядок и задаёт фазы историй: **Настройка → Еженедельный обход → Защита**.

## 2. User Stories с критериями приёмки

Формат: `As a / I want / So that` + **Given/When/Then** (критерии приёмки = будущие тесты, занятие 11) + трассировка + MoSCoW.

### Фаза A — Настройка (one-time, предусловие сценария)

#### US-A0. Войти в дашборд
**Как** преподаватель, **хочу** войти в систему по логину и паролю (или через SSO организации), **чтобы** получить доступ к дашборду своих курсов.
**Приоритет:** Must
**Критерии приёмки:**
- [ ] Дано: неаутентифицированный пользователь; когда я ввожу корректные логин и пароль; тогда я попадаю на главный экран дашборда (или выбор курса).
- [ ] Дано: я забыл пароль; когда нажимаю «Восстановить пароль»; тогда система отправляет ссылку для сброса на привязанную почту.
- [ ] Дано: я ввёл неверный пароль 5 раз подряд; тогда аккаунт временно блокируется на 15 минут.
**Трассировка:** Функциональное ядро — точка входа.

#### US-A1. Загрузить список репозиториев группы
**Как** преподаватель, **хочу** одним действием загрузить все репозитории группы (CSV со списком URL) и получить подтверждение, что они доступны, **чтобы** не добавлять 25 репо по одному и не выяснять на занятии, что репозиторий недоступен.
**Приоритет:** Must
**Критерии приёмки (основной поток — CSV):**
- [ ] Дано: пустой дашборд; когда я загружаю CSV с колонкой `repo_url` (опционально `branch`); тогда дашборд проверяет доступность каждого и показывает сводку: N доступно / M недоступно / K дубликатов URL.
- [ ] Дано: в CSV есть недоступный репо (нет прав / не существует); тогда дашборд помечает его как «недоступен», но остальные обрабатывает.
- [ ] Дано: в CSV есть дубликаты URL; тогда дашборд показывает предупреждение и дедуплицирует автоматически.
**Критерии приёмки (альтернативный поток — по одному):**
- [ ] Дано: пустой дашборд; когда я ввожу URL репозитория (опционально ветку); тогда он добавляется в список. Можно повторить N раз.
- [ ] Дано: в списке уже есть репозитории; когда я добавляю ещё один; тогда он дописывается, не сбрасывая уже добавленные.
**Критерии приёмки (завершение настройки):**
- [ ] Дано: список репозиториев загружен; когда я нажимаю «Готово»; тогда дашборд переходит в режим мониторинга (Фаза B).
- [ ] Дано: режим мониторинга активен; когда нужно добавить опоздавшего; тогда я возвращаюсь в настройку и добавляю репозиторий без потери уже собранной истории.
**Трассировка:** JTBD J3 ← боль «обхожу по одному, на всех не хватает».

#### US-A2. Настроить чек-лист артефактов по занятиям
**Как** преподаватель, **хочу** задать эталонный список ожидаемых артефактов по каждому занятию (имена файлов / этапы), **чтобы** инструмент знал, что считать «сдано».
**Приоритет:** Must
**Критерии приёмки:**
- [ ] Дано: конфиг чек-листа; когда добавляю занятие N с файлами `[a.md, b.md]`; тогда оно появляется столбцом в матрице.
- [ ] Дано: чек-лист задан; когда меняю состав файлов занятия; тогда статусы пересчитываются без переподключения репозиториев.
**Трассировка:** Функциональное ядро ← «источник правды о задачах — чек-лист курса».

#### US-A3. Подключить доступ к Git-платформе
**Как** преподаватель, **хочу** один раз настроить токен/OAuth к GitLab/GitHub, **чтобы** инструмент читал и приватные репозитории.
**Приоритет:** Must
**Критерии приёмки:**
- [ ] Дано: валидный токен; когда сохраняю; тогда дашборд подтверждает доступ и показывает время последней успешной синхронизации.
- [ ] Дано: токен протух; когда дашборд не может читать; тогда он показывает явную ошибку «обнови токен», а не молча показывает устаревшие данные.
**Трассировка:** Функциональное ядро.

### Фаза B — Еженедельный обход (ядро сценария)

#### US-B1. Картина группы за минуту
**Как** преподаватель, **хочу** видеть матрицу «репозиторий × занятие» со статусами наличия артефактов (есть / нет / частично), **чтобы** за минуту охватить всю группу перед занятием.
**Приоритет:** Must *(но это поверхность входа, не ядро ценности — см. US-B2)*
**Критерии приёмки:**
- [ ] Дано: 25 репозиториев и чек-лист на 5 занятий; когда открываю дашборд; тогда вижу матрицу 25×5 со статусом каждой ячейки.
- [ ] Дано: матрица; когда у студента артефакт есть, но лежит не в ожидаемой папке; тогда статус — «частично / не там», а не «нет» *(боль куратора Елены: «файл есть, но в другом месте»)*.
**Трассировка:** JTBD J3 ← «картины группы у меня нет нигде, она в голове и устаревает к четвергу».

#### US-B2. Подсветка разрывов связности — ЯДРО
**Как** преподаватель, **хочу** видеть, где в репозитории артефакт рвёт логическую цепочку (есть файл, но требования не бьются со схемой данных / архитектура не из плана), **чтобы** вмешаться точечно, а не перечитывать 18 файлов подряд.
**Приоритет:** **Must** — это главная ценность инструмента, а не «доработаем потом».
**Критерии приёмки:**
- [ ] Дано: репозиторий, где все файлы на месте; когда схема данных не ссылается на сущности из требований этого же студента; тогда дашборд помечает звено цепочки как «разрыв связности», а не зелёной галочкой.
- [ ] Дано: подсвечен разрыв; когда открываю его; тогда вижу, *между какими двумя артефактами* порвалась связь (а вчитываться в них Виталий идёт сам — машина только находит место).
**Трассировка:** JTBD J1 ← ключевой инсайт интервью: «файлы есть — смысла нет… отставание прощаю, разрыв смысла — нет».
**Операционализация (ADR-001, 2026-07-02 — supersedes прежний сплит FR-5a/FR-5b):** ядро v1 = **LLM-агент по версионированной рубрике**: сущности A получают статус в B (отражена / исключена явно / потеряна), разрыв = ≥1 потерянная; структурированный выход с точками-цитатами; вердикт привязан к `(sha_A, sha_B, рубрика, модель)`; тестируемость — golden set + метрика ≤20% ложных + override FR-10. Закладка ядра — за гейтом живого custdev (Фаза 0).
> 🎓 **Учебный момент (исторический):** в *первом черновике* prd.md эта история была в «Could / Версия 2», а `project-context` решением №3 объявлял сроки главной болью — оба теперь исправлены. Сам этот дрейф «инсайт → удобный симптом» разбираем на занятии как типовой дефект (см. `08-типовые-дефекты-prd.md`).

#### US-B3. Слепая зона: кто вне обзора инструмента
**Как** преподаватель, **хочу** видеть отдельным блоком студентов, чьи репозитории не добавлены в дашборд или недоступны, **чтобы** знать, кто вообще вне моего поля зрения, и написать им в чат.
**Приоритет:** Should
**Критерии приёмки:**
- [ ] Дано: студент есть в реестре, но его репозиторий не добавлен или недоступен при обходе; тогда студент выделен в блоке «вне обзора».
- [ ] Дано: репозиторий снова доступен; когда проходит обход; тогда студент автоматически уходит из блока «вне обзора».
**Трассировка:** JTBD J1 ← «смотрю не туда, где проблема, а туда, куда дотянулся» (кого не открыл — не видел).
> 🔧 **Синхронизация 2026-07-02 (найдено агентной проверкой ребра, golden set GS-1):** прежняя формулировка US-B3 смешивала два разных сигнала — «инструмент физически не видит репозиторий» и «репозиторий виден, но в нём тишина N занятий». Первый — это FR-6 (слепая зона), второй — это US-B4/FR-7 («хроники», ≥2 занятия без артефактов). Теперь US-B3 = только невидимость; сигнал тишины целиком живёт в US-B4.

#### US-B4. Хронически отстающий до выпадения
**Как** преподаватель, **хочу** видеть репозитории, отстающие по сдаче несколько занятий подряд, **чтобы** поймать «тихого» студента на 3-м занятии, а не на 12-м.
**Приоритет:** Should
**Критерии приёмки:**
- [ ] Дано: история сдач; когда студент отстаёт ≥2 занятий подряд; тогда он попадает в список «риск выпадения», независимо от активности в чате.
**Трассировка:** JTBD J1 ← «ловлю по тишине в чате — это не метод, это гадание».

#### US-B5. Данные актуальны и читаются сами
**Как** преподаватель, **хочу** видеть время последней синхронизации и кнопку «обновить сейчас», при этом данные тянутся из репо **без моего ручного ввода**, **чтобы** инструмент не «умер за три недели», как гугл-таблица.
**Приоритет:** Must *(жёсткое ограничение, не фича — см. NFR/бизнес-правило)*
**Критерии приёмки:**
- [ ] Дано: студент открыл новый MR; когда проходит синхронизация (по расписанию или по кнопке); тогда матрица обновляется без единого ручного ввода со стороны преподавателя.
- [ ] Дано: дашборд; когда смотрю на любой статус; тогда вижу, на какой момент времени он актуален.
**Трассировка:** JTBD J3 ← «любой инструмент, в который надо вводить руками, у меня умирает за три недели».

#### US-B6. Отметить ложный разрыв (override)
**Как** преподаватель, **хочу** пометить подсвеченный разрыв как ложный (синоним, переименование сущности), **чтобы** инструмент не дёргал меня той же ложной сработкой каждую неделю и доверие к подсветке не разрушалось.
**Приоритет:** Must *(добавлена по ревью 2026-06-30, дефект 10: FR-10 был без user story)*
**Критерии приёмки:**
- [ ] Дано: подсвеченный разрыв A→B; когда я помечаю его «ложный» (опционально — причина: синоним/переименование); тогда в текущем и последующих обходах этот разрыв не подсвечивается, пока пара артефактов не изменилась содержательно.
- [ ] Дано: разрыв помечен ложным; когда артефакт A или B изменился (новый `source_commit_sha`); тогда ребро проверяется заново — отметка не «замораживает» пару навсегда.
- [ ] Дано: разрыв помечен ложным по ошибке; когда я снимаю отметку; тогда разрыв снова виден в текущем обходе.
- [ ] Это исключение из BR-1: курирование находки инструмента, а не ввод данных о сдачах.
**Трассировка:** производная от FR-5a + BR-1 ← риск «ложные срабатывания подрывают доверие» (продуктовое ревью D6). Не самостоятельная потребность из интервью — помечена как производное требование.

### Фаза C — Защита

#### US-C1. Доказательная база цепочки на защите
**Как** преподаватель, **хочу** на защите за секунды показать всю цепочку артефактов студента с отметкой, *где* порвалась связность, **чтобы** обосновать оценку, а не выглядеть придирающимся.
**Приоритет:** Must
**Критерии приёмки:**
- [ ] Дано: студент говорит «я всё сдал вовремя»; когда открываю его репозиторий в дашборде; тогда вижу хронологию артефактов по занятиям с явной отметкой звена-разрыва.
- [ ] Дано: разрыв показан; когда студент оспаривает; тогда я могу указать на конкретные два артефакта, между которыми порвалась логика, не перечитывая все 18 файлов.
- [ ] Дано: студент сделал `force-push`/переписал историю репозитория; когда открываю хронологию на защите; тогда она остаётся корректной (привязка к `source_commit_sha`), а не рассыпается *(негативный сценарий из тех-ревью)*.
**Трассировка:** JTBD J2 ← «не могу ткнуть пальцем, ВОТ ЗДЕСЬ порвалось… выгляжу как препод, который придирается. Хотя я прав».

- **J1** — *Когда* готовлюсь к занятию, *хочу* видеть, где порвалась связность артефактов, *чтобы* вмешаться точечно. (ядро)
