# PRD: Course Dashboard — дашборд связности артефактов студенческих репозиториев

**Дата:** 2026-07-07 · **Версия:** v2.4 (правка §7 по В10: адрес репозитория — не ПДн) · **Автор:** ИИ + Виталий Генаров (преподаватель курса; он же владелец продукта — в тексте «CEO»)
**Статус:** продуктовые вопросы закрыты; перед разработкой ядра остался один гейт — Фаза 0 (§13). Технологии (стек, архитектура, выбор LLM-модели) — **сознательно вне этого документа**: они не меняют требований и решаются на этапе архитектуры (`../ARCHITECTURE.md` + ADR-002).

## 0. Контекст и как читать этот документ

**Откуда взялся продукт.** Идёт 10-недельный курс «Управление агентной разработкой»: ~25 студентов в одном учебном потоке (но часть сдает на GitLab, а другая - на GitHub), каждый ведёт собственный проект в публичном git-репозитории и после каждого занятия выкладывает туда документы-результаты (~18 штук за курс). Преподаватель один. Этот продукт — его личный инструмент контроля. Одновременно продукт — сквозной учебный пример: он сам разрабатывается теми методами, которым учит курс.

**Суть одним абзацем.** Инструмент сам читает документы из репозиториев студентов, показывает преподавателю таблицу «кто что сдал» и — главное — подсвечивает места, где у студента **порвалась логика между документами** (например, схема данных не использует ни одной сущности из его же требований). Преподаватель ничего не вводит руками и смотрит только в подсвеченные точки.

**Словарь терминов:**

| Термин | Значение |
|--------|----------|
| **Артефакт** | Документ (обычно `.md`-файл) с результатом этапа: интервью, персоны, user stories, PRD, схема данных, архитектура, план, код, тесты |
| **Связность** | Свойство портфолио: каждый следующий артефакт опирается на предыдущий. Главный критерий зачёта на курсе — важнее сроков сдачи |
| **Ребро связности** | Проверяемая пара «источник → приёмник», напр. «PRD → схема данных» |
| **Разрыв** | Значимая мысль источника **молча потеряна** приёмником (не отражена и не исключена явно) |
| **Рубрика** | Текстовое правило для LLM-агента, хранится в конфиге, имеет версию. Два вида: рубрика **ребра** (проверка связности пары документов, FR-5) и рубрика **шага** (проверка качества реализации одного документа, FR-11 — на вырост, v2) |
| **Обход (синк)** | Автоматический проход инструмента по всем репозиториям с чтением артефактов |
| **Слепая зона** | Репозитории, которые инструмент физически не видит: не добавлены или недоступны |
| **«Хроники»** | Студенты без новых артефактов 2+ занятия подряд (репозиторий виден, но в нём тишина) |
| **Отметка ложного разрыва (override)** | Решение преподавателя скрыть ошибочную находку агента (например, синоним принят за потерю) |
| **Golden set** | Набор эталонных пар документов с заранее известным вердиктом; регрессионная проверка ядра (`../evals/golden-set.md`) |
| **Фаза 0** | Блокирующий этап: проверка гипотезы ядра живым интервью до начала его разработки |
| **US / FR / NFR / BR / R-PD** | Пользовательская история / функциональное требование / нефункциональное требование / бизнес-правило / требование по персональным данным |
| **CEO** | Виталий Генаров — владелец продукта, он же преподаватель-пользователь (совмещение ролей намеренное: продукт делается для себя). Пометки «решено CEO» — его зафиксированные продуктовые решения |

**Связанные документы:** пользовательские истории — `user-stories/01-user-stories-преподаватель.md`; задачи пользователя — `jtbd.md`; интервью (синтетические) и персоны — `interviews/`, `personas/`; ключевые решения — `../decisions/` (ADR-001 — ядро); отчёты ревью этого PRD — `../reviews/`; история версий — `../archive/`.

> **Честность данных:** все интервью в основе этого PRD — **синтетические** (ИИ играл персону). Поэтому каждое требование — приоритизированная гипотеза, а не подтверждённый факт; гейт живой проверки вшит в §13.

## 1. Проблема

Преподаватель ведёт 25 студентов × ~18 артефактов = ~450 документов за курс. Критерий зачёта — связность, а не только наличие файлов.

**Сегодня без инструмента:** ручной выборочный обход репозиториев по воскресеньям — успевает открыть 8–10 из 25; остальное — слепая зона. Отстающих ловит по «тишине в чате», то есть гаданием. На защите проекта видит «кашу» в портфолио, но не может показать, *где именно* порвалась логика, не перечитав все 18 файлов.

**Ключевой сигнал интервью (гипотеза о ядре):**
> «Отставание по срокам прощаю. Разрыв смысла — нет… Найди мне разрывы — читать буду я.»

→ Ядро продукта — **подсветка разрывов связности**, а не таблица «сдал/не сдал». *(Сигнал синтетический; до разработки ядра обязательна живая проверка — Фаза 0.)*

**Цена бездействия:** при росте до 2–3 потоков «память + воскресенье» рушится → курс деградирует до сдачи файлов ради галочек → падает доля дошедших до защиты.

## 2. Решение

Инструмент по расписанию читает `.md`-файлы из публичных репозиториев студентов (GitLab/GitHub API), сверяет с эталонным чек-листом артефактов по занятиям и строит дашборд: **матрица «репозиторий × занятие» плюс подсвеченные разрывы связности**. Преподаватель видит группу на одном экране за ≤5 минут перед занятием, проваливается в точки разрыва с цитатами, на защите за секунды показывает доказательную цепочку. Данные читаются автоматически; преподаватель только курирует находки (может отметить ложный разрыв).

**Пользовательский поток.** Настройка (один раз): вход → список репозиториев (CSV) → чек-лист артефактов + рёбра связности → токен доступа к Git API. → **Дважды в неделю (перед занятиями):** матрица группы → подсвеченные разрывы → слепая зона и «хроники». → **Защита:** карточка студента → хронологическая цепочка артефактов с местами разрывов.

## 3. Аудитория

**v1 — однопользовательский инструмент: только преподаватель курса.** Закрыт аутентификацией, без студенческого и публичного интерфейса.

- **Пользователь:** преподаватель — главный ЛПР (лицо, принимающее решение) и единственный, кто открывает дашборд.
- **Стороны влияния (накладывают ограничения, но НЕ пользователи):** служба безопасности (закон о персональных данных 152-ФЗ), администрация (следит за долей дошедших до защиты), куратор и другие преподаватели — кандидаты будущих версий.
- **Намеренно НЕ пользователи:** студенты. Дашборд ничего им не показывает и не сравнивает их между собой (BR-4).

## 4. Цели и метрики

| Цель | Метрика | Целевое значение |
|------|---------|------------------|
| Снять ручную механику сверки | Время обзора группы перед занятием | ≤ 5 мин |
| Покрыть всю группу | Доля репозиториев вне слепой зоны | 25/25 |
| Исключить ручной ввод | Действий преподавателя по вводу данных в неделю | 0 |
| Доказательная база на защите | Время найти место разрыва на защите | секунды |
| **Доверие к ядру (FR-5)** | Доля ложных разрывов среди подсвеченных | ≤ 20% на пилоте — иначе ядро не проходит Фазу 0; регрессии ловятся по golden set |

> **Слепое пятно метрики (находка ревью):** доля ложных срабатываний не ловит **пропуски** — случаи, когда агент счёл мысль «отражённой» по поверхностному совпадению, а реальный разрыв прошёл незамеченным. На пилоте Фазы 0 обязательна ручная сверка полноты на 3 репозиториях: сколько реальных разрывов поймано, сколько пропущено. Найденные пропуски пополняют golden set.

### 5.1 Пример работы ядра (иллюстрация к FR-5; канон — ADR-001 и `../evals/golden-set.md`)

**Рубрика ребра «PRD → схема данных» (сокращённо):** «Схема данных обязана опираться на PRD: каждая значимая сущность PRD (объект, о котором система должна помнить) либо отражена в схеме (синонимы засчитываются и называются), либо явно исключена с обоснованием, либо потеряна. Разрыв = ≥1 потерянная сущность».

**Пример структурированного выхода агента:**

```
ВЕРДИКТ: РАЗРЫВ · УВЕРЕННОСТЬ: высокая
СУЩНОСТЕЙ ПРОВЕРЕНО: 12 · отражено 10 · исключено явно 1 · потеряно 1
ТОЧКА 1: «отметка ложного разрыва» | цитата PRD: «отметка обратима, не замораживает
  пару навсегда» (FR-10) | в схеме нет таблицы или полей для отметок — искал
  «отметка/override/ложный» и синонимы, не нашёл и в разделе исключений
ЗАМЕТКА: проверь, где схема хранит решения преподавателя по находкам агента.
```

Преподаватель видит только такие карточки-точки; вердикт «как оценивать студента» остаётся за ним (BR-2).

### 5.2 На вырост — v2: анализ качества реализации артефактов (FR-11)

**Идея.** v1 отвечает на вопросы «сдано ли?» (FR-4) и «связно ли?» (FR-5). v2 добавляет третий: **«сделан ли документ как учили?»** У каждого шага курса есть канон — то, чему учило занятие: у PRD — проблема с ценой бездействия, измеримые метрики, out-of-scope, помеченные допущения; у схемы данных — трассировка сущностей к требованиям и обоснованное решение «история vs состояние»; и так далее. FR-11 превращает канон каждого шага в **версионированную рубрику шага** и прогоняет каждый артефакт по скрипту через LLM.

**Как работает (тот же паттерн, что ядро FR-5):** скрипт берёт документ + рубрику его роли → агент возвращает структурированную карточку: покрытие пунктов рубрики (выполнен / частично / отсутствует), до 5 замечаний с цитатами из документа, заметку преподавателю. Результат привязан к `(версия документа, версия рубрики, модель)` и не пересчитывается, пока они не изменились; регрессии ловятся golden set-ом; ложное замечание преподаватель гасит отметкой (аналог FR-10).

**Граница (BR-2 сохраняется и в v2):** выход FR-11 — **не балл, не оценка и не рейтинг**, а карточка соответствия рубрике с цитатами. Оценивает студента по-прежнему человек; инструмент показывает, куда смотреть. Межстуденческого сравнения качества не появляется (BR-4).

**Вход в v2 (что нужно до реализации):** пользовательская история с критериями приёмки (производных требований без US не заводим — урок FR-10); рубрики шагов как артефакты курса (по одной на занятие — их естественно писать вместе с планами занятий); пилот точности на артефактах прошедшего потока. Корень уже заложен в курсовом решении ADR-003: «агент сверяет артефакты с чек-листом и оценивает прозу по рубрике → дайджест преподавателю».

## 6. Бизнес-правила и ограничения

- **BR-1 (ноль ручного ввода):** преподаватель не вводит данные о сдачах руками — они читаются из репозиториев. Единственное исключение — курирование находок (FR-10). Загрузка CSV со списком репозиториев — разовая настройка, не ввод данных сдач.
- **BR-2 (разделение труда):** агент находит и подсвечивает *точки* разрыва с цитатами и объяснением; преподаватель смотрит только в подсвеченные точки и сам выносит вердикт об оценке. Инструмент не оценивает качество текста и не ставит оценок.
- **BR-3 (статус «частично» — решено CEO 2026-07-02):** «частично» = неточное имя/место ИЛИ файл-заготовка (пустой / не изменён относительно шаблона). Файл не в ожидаемой папке — «частично/не там», а не «нет». Порог минимального объёма текста НЕ вводим: короткий хороший файл — не дефект.
- **BR-4 (приватность):** v1 виден только преподавателю. У студентов нет доступа; сравнения студентов между собой не существует как экрана. Конфликт «контроль преподавателя ↔ страх студента перед „табло позора"» снят радикально — отсутствием студенческого интерфейса.
- **BR-5 (публичность репозиториев):** репозиторий студента — публичный (курсовое решение ADR-003). Курс идёт двумя учебными потоками на разных хостингах — один на GitLab, другой на GitHub (см. §0); публичность — единственный механизм доступа, работающий одинаково для обоих. Токен нужен из-за лимитов API, а не для доступа к приватному.
- **BR-6 (реестр студентов):** список «ФИО ↔ адрес репозитория» ведётся во внешней форме (Яндекс.Форма) и публикуется как CSV для автоматического чтения (ADR-003; способ выгрузки — открытый вопрос №7).

## 7. Персональные данные (152-ФЗ) — требования, а не декларация

> Продукт приватный, публичного вида нет — главный риск раскрытия данных снят. Но именные данные студентов обрабатываются (легитимно, как журнал оценок), поэтому требования остаются. Адрес репозитория (в т.ч. содержащийся в нём логин) персональными данными **не считается** — решено CEO 2026-07-02, подтверждено 2026-07-07; ПДн в продукте — именной реестр «ФИО ↔ адрес репозитория».

- **R-PD1 (доступ):** дашборд закрыт аутентификацией (FR-0); доступ только у преподавателя.
- **R-PD2 (токен):** только чтение, шифрование при хранении, не в коде и не в логах (NFR-3).
- **R-PD3 (периметр обработки — рамка CEO 2026-07-02):** развёртывание допустимо (нужен автообход), но **максимум данных остаётся у преподавателя**: именной реестр «ФИО ↔ адрес» не покидает его контур без необходимости. В LLM-сервис уходит только содержимое **публичных** репозиториев и текст рубрик — именной реестр не отправляется (ADR-001). Точное разделение компонентов — на этапе архитектуры.
- **R-PD4 (хранение и удаление — решено CEO 2026-07-02):** именные данные удаляются **через 30 дней после защиты** (окно на апелляции и пересдачи). Обезличенная статистика работы инструмента (число разрывов, точность подсветки) может храниться дольше.

## 8. Пользовательские истории

Канонический файл: `user-stories/01-user-stories-преподаватель.md` — истории US-A0…US-C1 (+ US-B6) в формате Given/When/Then с матрицей трассировки к задачам и цитатам интервью. PRD истории **не дублирует** — чтобы не появлялось двух копий правды, которые неизбежно разъезжаются. Сквозной сценарий: настройка → обход два раза в неделю → защита.

**Разделение канона:** детальные сценарные критерии (Given/When/Then) канонически живут в файле историй; таблица §5 — их сводка плюс скоуп и приоритеты. При расхождении сценарных деталей прав файл историй, при расхождении скоупа/приоритетов — PRD; любое найденное расхождение чинится в обоих местах с пометкой даты (прецедент — синхронизация US-B3 от 2026-07-02).

## 9. Не входит в v1 (явно НЕ делаем)

- **Любой студенческий и публичный интерфейс.** Сценарии студентов — Дамира (портфолио для работодателя) и Алии (приватный список «что осталось доделать») — возможные будущие отдельные продукты, не части этого дашборда.
- **Сигнал приёмки для студента** («я сдал — это принято?»). Паркуем осознанно: это сильнейший общий сигнал всех интервью, но он требует студенческого интерфейса, которого нет намеренно ни в v1, ни в v2. v1 ценен и без него: он целиком закрывает задачи преподавателя (J1–J3), не касаясь задач студентов. Возвращаемся не раньше v3.
- **Анализ качества реализации артефактов** — не v1, а содержание **v2** (FR-11, §5.2). В v1 — только наличие (FR-4) и связность (FR-5). При этом «любые оценки/баллы от ИИ» остаются за границей BR-2 во всех версиях: и FR-5, и FR-11 подсвечивают и цитируют, а не оценивают.
- Доступ куратора и других преподавателей; сводка по нескольким потокам; аналитика удержания; уведомления студентам; мобильная версия и Telegram-интерфейс.

## 11. Риски

| Риск | Влияние | Снижение |
|------|---------|----------|
| Ядро даёт ложные разрывы или пропускает реальные | Подрыв доверия с первой же ошибки | Отметка ложного разрыва (FR-10) + метрика §4 + golden set + гейт Фазы 0 |
| Вердикт LLM меняется от прогона к прогону | «Мигание» разрывов; особенно опасно на защите | Вердикт привязан к версии документов, рубрики и модели; пересчёт только при их изменении (ADR-001) |
| LLM-сервис недоступен или дорожает | Ядро временно не работает | Матрица и хронология работают без LLM; проверки «отложены»; объём мал (~200 пар/неделю) |
| Ложный разрыв показан **на защите** — цена ошибки выше, чем в будничном обходе | Несправедливая оценка, конфликт при комиссии | На защите флаг разрыва — только при уверенности «высокая» (шкала FR-5) и если не отменён преподавателем; хронология артефактов показывается всегда |
| Подсветку агента примут за оценку качества | Завышенное обещание, конфликт на защите | Граница BR-2: агент подсвечивает точки и цитаты, оценивает человек |
| Файлы-заготовки из шаблона засчитаны как «сдано» | Ложная полнота матрицы | Закрыто правилом: заготовка = «частично» (FR-4, BR-3) |
| Именные данные студентов | Претензии по обработке персональных данных | §7: закрытый доступ, минимизация, удаление через 30 дней |
| Преподаватель проваливается в чтение всего подряд | Инструмент не экономит время | Интерфейс ведёт сразу в подсвеченные точки (FR-5 поверх FR-4) |

## 13. Этапы и заметки по внедрению

- **Фаза 0 — БЛОКИРУЮЩИЙ гейт ядра:** ≥1 **живое** интервью с преподавателем (например, коллегой с другого курса) + прогон рубрики на 1–3 чужих реальных репозиториях. Гейт запрещает **закладывать ядро в код продукта**, но не запрещает проверять само правило: 2026-07-02 выполнен ручной прогон рубрики через LLM на 2 парах документов собственного проекта — без какого-либо кода (`../evals/golden-set.md`). Он подтвердил исполнимость правила; ценность для живого пользователя по-прежнему гипотеза до Фазы 0. *(Периферию — FR-0..4, 8, хронологию FR-9 — Фаза 0 не блокирует.)*
- **v1 (прототип):** FR-0..FR-5, FR-8, FR-9, FR-10 + R-PD1/R-PD2 — полный сценарий «настройка → обход → защита».
- **v1.1:** FR-6, FR-7 (слепая зона, «хроники»).
- **v2 (решение CEO 2026-07-02):** **FR-11 — анализ качества реализации артефактов каждого шага** по версионированным рубрикам шагов (§5.2). Вход в v2: US с критериями + рубрики шагов как артефакты курса + пилот точности.
- **v3 и дальше:** сигнал приёмки для студентов, сценарии Дамира/Алии, мультикурс.
- **Модель данных — история наблюдений, не «текущее состояние»:** статусы хранятся по занятиям/обходам со ссылкой на коммит (`source_commit_sha`) — иначе невыполнимы «хроники» (динамика) и хронология на защите, устойчивая к переписыванию истории репозитория. Логическая схема — артефакт занятия 6 (`../data-model.md`).
- **Порядок тестирования — сначала негативные сценарии:** лимиты API, протухший токен, приватный/пустой репозиторий, файл-заготовка из шаблона, ветка по умолчанию ≠ `main`, усечённые ответы API, переименованный файл (→ «частично»), переписанная история (`force-push`).
- **Подход к разработке:** сначала критерии приёмки конкретного требования, потом код; код — производное от критериев.
