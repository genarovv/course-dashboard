# PRD: Дашборд проверки работ студентов курса «Управление агентной разработкой»

**Дата:** 2026-06-30 · **Статус:** Draft · **Автор:** AI + Виталий
**Версия:** v1

---

## 1. Формулировка проблемы

25 студентов × ~18 артефактов = ~450 `.md`-файлов, которые преподаватель проверяет ручным выборочным обходом репозиториев по выходным — это не масштабируется, оставляет слепые зоны (8–10 из 25), а на защите проектов нет доказательной базы для оценки связности артефактов.

## 2. Решение

Когда дашборд реализован, преподаватель видит всю группу на одном экране за ≤5 минут перед занятием: матрица наличия артефактов по занятиям, подсвеченные разрывы связности между артефактами каждого студента, слепые зоны (кто не выложил репозиторий) и хронически отстающие (≥2 занятий подряд). На защите проекта преподаватель за секунды находит и показывает цепочку разрыва — от требования к схеме данных, от схемы к архитектуре. Все данные читаются автоматически из публичных репозиториев студентов по Git API; преподаватель не вводит ничего вручную, только курирует находки (отмечает ложный разрыв).

## 3. Аудитория

**v1 — однопользовательский инструмент: только преподаватель курса.** Закрыт аутентификацией, без студенческого и публичного интерфейса.

- **Пользователь:** преподаватель (главный ЛПР и единственный, кто открывает дашборд)
- **Стороны влияния (накладывают ограничения, но НЕ пользователи v1):** служба безопасности (152-ФЗ), администрация (метрика «% дошедших»)
- **Намеренно НЕ пользователи:** студенты. Дашборд ничего им не показывает и не сравнивает их между собой.

## 4. Цели и метрики

| Цель | Метрика | Целевое значение |
|------|---------|------------------|
| Снять ручную механику сверки | Время обзора группы перед занятием | ≤ 5 мин |
| Покрыть всю группу | Доля репозиториев вне слепой зоны | 25/25 |
| Исключить ручной ввод | Действий преподавателя по вводу данных в неделю | 0 |
| Доказательная база на защите | Время найти место разрыва на защите | секунды |
| Доверие к подсветке разрывов | Доля ложных разрывов среди подсвеченных | ≤ 20% на пилоте |

## 5. Функциональные требования и критерии приемки

| ID | Требование | Критерий приемки | Приоритет |
|----|------------|-------------------|:---------:|
| FR-1 | Подключение репозиториев группы | Преподаватель добавляет список URL репозиториев группы за один приём (групповой импорт / форма). После добавления дашборд начинает читать артефакты. | P0 |
| FR-2 | Чек-лист артефактов и граф рёбер связности | Конфиг определяет ожидаемые артефакты по занятиям + пары «A→B» для проверки связности (напр. `требования → схема данных`). Конфиг читается из файла (не хардкод). | P0 |
| FR-3 | Доступ к Git API | Дашборд авторизуется по токену (GitLab/GitHub), индицирует протухание токена, сообщает об ошибках доступа к конкретному репозиторию (не валит весь обход). | P0 |
| FR-4 | Матрица «репозиторий × занятие» | Для каждого студента отображается: какие артефакты сданы (есть/нет/частично) по каждому занятию. Частично = файл найден не в ожидаемом месте или имени. | P0 |
| FR-5a | Подсветка структурного разрыва связности | Для каждого ребра A→B из FR-2 проверяется: хотя бы одна именованная сущность из A встречается в B. Не встречается ни одна — разрыв подсвечен. Лексическое перекрытие, НЕ семантика. | P0 |
| FR-6 | Выделение слепой зоны | Студенты, чьи репозитории не добавлены или недоступны — вынесены в отдельный блок «вне обзора». | P1 |
| FR-7 | Хронически отстающие | Студенты с отсутствием артефактов ≥2 занятия подряд — выделены в группу «хроники». | P1 |
| FR-8 | Автосинхронизация | Дашборд синхронизируется по расписанию (напр. ежедневно) или по запросу. Инкрементально: не перечитывает всё каждый раз. Пользователь видит статус «идёт синк / актуально на ЧЧ:ММ». | P0 |
| FR-9 | Доказательная цепочка на защите | Преподаватель открывает карточку студента → видит разрывы в хронологическом порядке → может «развернуть» разрыв с цитатами из A и B (какая сущность объявлена, какая не найдена). Хронология устойчива к force-push (time-series). | P0 |
| FR-10 | Отметка ложного разрыва (override) | Преподаватель может пометить разрыв как ложный (синоним, переименование). Это не ручной ввод данных, а курирование результата. Отмеченные разрывы не подсвечиваются в следующих обходах. | P0 |
| NFR-1 | Производительность | Полный обход 25 репозиториев × ~18 файлов — не более 2 минут. | P0 |
| NFR-2 | Graceful degradation | Частичный успех (один репозиторий недоступен) не валит весь дашборд. | P0 |
| NFR-3 | Безопасность токена | Git-токен хранится зашифрованно, не в репозитории кода. | P0 |
| NFR-4 | Rate-limit / backoff | Корректная обработка лимитов Git API с паузами и повторными попытками. | P0 |

## 6. Бизнес-правила и ограничения

- **BR-1 (отсутствие ручного ввода):** преподаватель не вводит данные о сдачах руками — они читаются из репозитория. Допустима только отметка ложного разрыва (FR-10), которая не является вводом данных, а курированием результата.
- **BR-2 (разделение труда):** машина находит *структурные* разрывы (лексическое перекрытие); вчитывание в смысл и оценка качества — за преподавателем. Инструмент подсвечивает место, не выносит вердикт.
- **BR-3 (статус «частично»):** артефакт не в ожидаемой папке/имени — статус «частично/не там», а не «нет». Правила матчинга задаются в конфиге (FR-2).
- **BR-4 (приватность):** v1 — только teacher-facing. У студентов нет доступа к дашборду. Межстуденческого сравнения не существует как поверхности.
- **BR-5 (публичность репозиториев):** репозиторий студента — публичный. Это единственный механизм, одинаковый для GitLab и GitHub потоков.
- **BR-6 (реестр студентов):** реестр с URL репозиториев ведётся во внешней форме (Яндекс.Форма). Публикуется как CSV для автоматического чтения.

## 7. Пользовательские истории

- **US-1 (преподаватель, еженедельный обход):** В роли преподавателя в воскресенье вечером перед занятием, я хочу открыть дашборд и за 5 минут увидеть матрицу «кто что сдал» с подсветкой разрывов связности, чтобы перед занятием знать, у кого какие проблемы и кому нужно уделить внимание.
- **US-2 (преподаватель, глубокий просмотр):** В роли преподавателя увидев подсвеченный разрыв у студента, я хочу кликнуть на него и сразу увидеть: какой артефакт-источник (с сущностями), какой артефакт-приёмник (без них) и конкретно какие сущности не найдены, чтобы решить, это настоящий разрыв или ложный (синоним).
- **US-3 (преподаватель, защита проекта):** В роли преподавателя на защите проекта, я хочу за 10 секунд найти и показать студенту и комиссии цепочку разрыва между его PRD и схемой данных, чтобы аргументировать оценку связности портфолио.
- **US-4 (преподаватель, подключение потока):** В роли преподавателя в начале курса, я хочу загрузить CSV-реестр с URL всех репозиториев группы одним действием и получить первую матрицу в течение 2 минут, чтобы не вбивать 25 URL по одному.
- **US-5 (преподаватель, слепая зона):** В роли преподавателя перед занятием, я хочу видеть отдельный блок «не добавлены / недоступны» со списком студентов, чьи репозитории я не вижу, чтобы знать, кто вне моего обзора и написать им в чат.
- **US-6 (преподаватель, хроники отставания):** В роли преподавателя на 5-й неделе курса, я хочу видеть, кто из студентов не сдавал артефакты 2+ занятия подряд, чтобы своевременно провести интервенцию, а не обнаружить отсев на защите.

## 8. Не входит в план работ

- Любой студенческий и публичный интерфейс. Студенты не видят дашборд, не получают нотификации от него, не сравниваются между собой.
- Семантический LLM-детект «разрыва смысла» (FR-5b) — v2. v1 даёт только структурный прокси (лексическое перекрытие).
- Оценка **качества** содержания артефакта с помощью ИИ (есть/нет — да; плохо/хорошо — нет).
- Доступ куратора / другого преподавателя.
- Сводка по нескольким курсам или потокам на одном экране.
- Метрики retention, отсева, статистика по группе.
- Нотификации и авто-сообщения студентам.
- Мобильная версия / Telegram-интерфейс.

## 9. Допущения

| # | Допущение | Риск, если неверно | Как проверить |
|---|-----------|--------------------|---------------|
| A1 | Ядро ценности для преподавателя — подсветка разрывов связности, а не матрица «сдал/не сдал» | Строим не тот продукт (нужна была просто таблица) | 1 живое интервью с преподавателем другого потока + прогон прототипа |
| A2 | Структурный прокси (лексическое перекрытие) достаточно полезен как v1, до LLM-семантики | Ядро не доставляет ценность (ложных разрывов слишком много) | Прогон на 3 реальных репозиториях, замер доли ложных срабатываний |
| A3 | Преподаватель примет инструмент, который читает данные из репозиториев автоматически и не требует ручного ввода | Низкая адопция, возврат к «воскресному обходу» | Пилот на 1 потоке, наблюдение за поведением |
| A4 | Реестр студентов (Яндекс.Форма) может быть экспортирован в машиночитаемый формат | Агент-обходчик не может автоматически получить список URL | Настроить интеграцию формы с таблицей; fallback — ручной CSV от преподавателя |
| A5 | Студенты используют публичные репозитории (GitLab culab.ru или GitHub) с .md-артефактами | Часть репозиториев приватные / не в .md / другая структура | Зафиксировать в контракте артефактов (ADR-003) на старте курса |

## 10. Заметки по внедрению

- **Архитектура:** time-series хранение статусов (не snapshot). История по неделям со ссылкой на `source_commit_sha`. Это единственный способ выполнить US-3 (защита) и US-6 (хроники) — хронология не ломается при force-push.
- **MVP — минимально жизнеспособный сценарий:** FR-1, FR-2, FR-3, FR-4, FR-5a, FR-8, FR-9, FR-10. Это покрывает полный цикл «настройка → еженедельный обзор → защита». FR-6 и FR-7 — в следующую итерацию.
- **Конфиг рёбер связности (FR-2):** начальные рёбра для курса: `требования (PRD) → схема данных`, `схема данных → архитектура`, `архитектура → план разработки`, `user stories → PRD`, `интервью → персоны`, `персоны → user stories`. Каждое ребро проверяет: сущность из A встречается в B.
- **Стек:** веб-дашборд (любой фронтенд-фреймворк) + серверная часть (чтение Git API, парсинг .md, хранение time-series). LLM (FR-5b) не нужен для v1.
- **Реестр студентов:** реализовать чтение через экспорт Яндекс.Формы в CSV. Если авто-экспорт невозможен — предусмотреть ручную загрузку CSV преподавателем (fallback, не нарушающий BR-1, так как это импорт списка URL, а не ввод данных сдач).
- **Приоритет тестирования:** сначала негативные сценарии — rate-limit Git API, протухший токен, приватный репозиторий без прав, пустой репозиторий, default branch ≠ main, force-push. Затем — точность FR-5a на реальных данных.
- **Живая валидация (гейт перед реализацией):** ≥1 живое интервью с преподавателем + прогон FR-5a на 1–3 реальных репозиториях. Пока этого нет, ядро (разрывы связности) — гипотеза, а не подтверждённая потребность.
- **Подход к реализации:** spec-driven — перед любой генерацией кода описать acceptance criteria для конкретного FR / user story. Код — производное от acceptance criteria.
