# PRD — Course Dashboard (парсер репозиториев студентов + дашборд связности артефактов)

**Дата:** 2026-06-30 · **Статус:** v1, готов к декомпозиции (занятие 7) · **Автор:** ИИ + Виталий
**Вход:** `course/занятие-05-демо/01-user-stories-преподаватель.md` (сквозной сценарий преподавателя) ← синтетические интервью занятия 4.
**Тип пользователя v1:** преподаватель (главный ЛПР). Дамир/Алия — отдельные сценарии (out of scope v1).

> **Это эталон занятия 5.** Прошёл 4-линзовое ревью (трассировка / продукт / техника / аудитория) — отчёты `занятие-05-демо/05–07`. Главный урок ревью: *ядро можно объявить в прозе и тут же потерять в критериях приёмки и метриках.* Этот PRD operationalize ядро, а не только декларирует.

## 1. Проблема (зачем и почему сейчас)

Преподаватель ведёт 25 студентов × ~18 артефактов = ~450 `.md`-единиц. Критерий зачёта — **связность артефактов** (каждый этап опирается на предыдущий), а не наличие файлов.

**Сегодня без инструмента:** ручной выборочный обход репозиториев по воскресеньям (8–10 из 25), память и «тишина в чате» как индикатор. Две дыры: **слепая зона** (кого не открыл — не видел) и **нет доказательной базы** на защите (видит «кашу», но не покажет где, не перечитав 18 файлов).

**Ключевой сигнал синтетического интервью (гипотеза о ядре):**
> «Отставание по срокам прощаю. Разрыв смысла — нет… Найди мне разрывы — читать буду я.»

→ Ядро — предположительно **подсветка разрывов связности**, а не матрица «сдал/не сдал». *(Источник — синтетическое интервью; до закладки ядра обязателен живой custdev — см. §11, Фаза 0.)*

**Цена бездействия:** при росте до 2–3 потоков «память + воскресенье» рушится → курс деградирует до сдачи файлов ради галочек → падает % дошедших до защиты.

## 2. Аудитория

**Продукт v1 — однопользовательский: только преподаватель.** Приватный инструмент (за доступом), без студенческого и публичного интерфейса.

- **Единственный пользователь (v1):** преподаватель — главный пользователь и ЛПР (персона Виталий).
- **Стороны влияния (НЕ пользователи v1, но диктуют ограничения):** служба безопасности (152-ФЗ при обработке именных данных), администрация ЦУ (метрика «% дошедших» — инструмент не должен ей вредить), куратор и другой преподаватель — потенциальные пользователи будущих версий, не v1.
- **Намеренно НЕ пользователи:** студенты (Дамир, Алия). Дашборд их не показывает им и не сравнивает между собой — см. BR-4 и §8.

## 3. Цели и метрики успеха

| Цель | Метрика | Целевое значение |
|------|---------|------------------|
| Снять механику сверки | Время обзора группы перед занятием | ≤ 5 мин |
| Покрыть всю группу | Доля репозиториев вне слепой зоны | 25/25 |
| Инструмент выживает | Ручных вводов данных преподавателем в неделю | 0 |
| Доказательная база | Время найти место разрыва на защите | секунды |
| **Доверие к ядру (FR-5a)** | **Точность подсветки разрыва (доля ложных разрывов)** | **ложных ≤ 20% на пилоте — иначе FR-5a не проходит гейт §11** |

> **Правка по ревью:** прежняя метрика «отстающий замечен ≤ 3-го занятия» убрана — она недостижима скоупом v1 и тащит обратно дедлайн-симптом, от которого мы как раз уходим. Раннее обнаружение измеряем не по сроку, а по доле разрывов, разобранных **до** защиты.

## 4. Решение (верхнеуровнево)

Инструмент по расписанию читает `.md`-файлы из репозиториев студентов (GitLab/GitHub API), сверяет с эталонным чек-листом артефактов по занятиям и строит дашборд: матрица «репозиторий × занятие» **плюс подсветка структурных разрывов связности**. Преподаватель видит группу за минуту, проваливается в разрывы, на защите показывает доказательную цепочку.

### 4.1 Приоритизированные требования

| ID | Требование | US | JTBD | Приоритет |
|----|------------|----|----|:---------:|
| FR-1 | Подключение репозиториев группы списком/группой | US-A1 | J3 | P0 |
| FR-2 | Чек-лист артефактов **+ граф рёбер связности** в конфиге | US-A2 | ядро | P0 |
| FR-3 | Доступ к Git API (токен/OAuth) + индикация протухания | US-A3 | ядро | P0 |
| FR-4 | Матрица «репозиторий × занятие» (есть/нет/частично) | US-B1 | J3 | P0 (вход) |
| **FR-5a** | **Подсветка разрыва связности — структурный прокси (v1)** | **US-B2** | **J1** | **P0 (ядро)** |
| FR-5b | Семантический детект «разрыва смысла» — LLM | US-B2 | J1 | P2 (v2) |
| FR-6 | Выделение слепой зоны | US-B3 | J1 | P1 |
| FR-7 | Хронически отстающие (≥2 занятий подряд) | US-B4 | J1 | P1 |
| FR-8 | Автосинхронизация без ручного ввода + отметка актуальности | US-B5 | J3 | P0 |
| FR-9 | Доказательная цепочка на защите (← строится на FR-5a) | US-C1 | J2 | P0 |
| FR-10 | Отметка ложного разрыва (override) — на находке, не ввод данных | US-B2 | J1 | P0 |

### 4.2 Операционализация ядра (FR-5a) — честный критерий v1

> **Главное по ревью:** ядро нельзя оставлять словами «порвалась связь». Определяем проверяемо.

- **Ребро связности** — пара артефактов (A→B), заданная в конфиге чек-листа (FR-2): напр. `требования → схема данных`, `план → архитектура`.
- **«Сущность»** — именованный токен из словаря этапа A (имя таблицы/поля/модуля), извлекаемый по правилам конфига.
- **Структурный разрыв (v1)** = в артефакте B **не встречается ни одна** сущность, объявленная в A по этому ребру. Это **лексическое перекрытие словаря**, а НЕ оценка смысла.
- **Честная граница (произнести):** FR-5a ловит *структурный* сигнал-кандидат, а не семантику. Возможны ложные разрывы (синонимы, переименования) → обязателен FR-10 (override) и метрика точности (§3). Семантику обещает только FR-5b (v2).

### 4.3 Пользовательский поток
Настройка (один раз): репозитории → чек-лист+граф рёбер → токен. → **Еженедельно:** матрица группы → разрывы связности (с возможностью отметить ложный) → слепая зона/хроники. → **Защита:** репозиторий студента → цепочка с местом разрыва.

## 5. Технические требования

- **Стек/интеграции:** клиент GitLab/GitHub API · парсер `.md` · хранилище состояния · веб-дашборд.
- **Модель данных — time-series, не снапшот** (по ревью): статусы хранятся как история по занятиям/неделям со ссылкой на `source_commit_sha`, иначе невыполнимы Must-истории US-B4 (динамика) и US-C1 (хронология на защите устойчива к `force-push`). Полная схема — занятие 6.
- **Синхронизация (NFR):** инкрементальный синк 25×~18 файлов; **rate-limit/backoff** для Git API; **частичный успех** не валит весь дашборд (недоступный репо помечается, остальные обновляются); пользователь видит статус «идёт синк / актуально на ЧЧ:ММ».
- **Негативные/граничные сценарии (обязательны, по ревью):** rate-limit API, протухший токен, приватный репо без прав, пустой репо, монорепо/форк, `default branch ≠ main`, пагинация >N файлов, переименованный/перемещённый файл (→ статус «частично/не там», BR-3), `force-push`/переписанная история (хронология US-C1 не должна ломаться).

## 6. Бизнес-правила

- **BR-1 (выживаемость, уточнено):** преподаватель **не вводит данные о сдачах руками** — они читаются из репо. Допустимы лишь решения по находкам инструмента (отметка ложного разрыва FR-10) — это не ручной ввод данных, а курирование результата.
- **BR-2 (разделение труда):** машина находит *структурные* разрывы; вчитывание в смысл — за преподавателем. Инструмент подсвечивает место, не выносит вердикт о качестве.
- **BR-3 (статус «частично»):** артефакт не в ожидаемой папке/имени — «частично/не там», не «нет». Порог «частично» задаётся правилом сопоставления в конфиге (FR-2).
- **BR-4 (приватность — снята сужением скоупа):** продукт v1 **не student-facing** — у студентов нет доступа к дашборду, межстуденческого сравнения не существует как поверхности. Конфликт «контроль преподавателя ↔ "табло позора" Алии» снят **радикально, отсутствием студенческого интерфейса**, а не правилом видимости. Любой студенческий/публичный вид — out-of-scope (§8).

## 7. 152-ФЗ — требования, а не декларация (по ревью СБ)

> Раз продукт приватный (teacher-only), публичный вид не нужен — это упрощает периметр. Но именные данные студентов обрабатываются легитимно (как журнал оценок), значит требования по обработке остаются.

- **R-PD1 (доступ):** дашборд **закрыт аутентификацией**, доступ только у преподавателя. Никакого публичного/анонимного вида. Именная привязка прогресса к студенту допустима как часть законного учебного процесса.
- **R-PD2 (токен):** Git-токен хранится зашифрованным, не в репозитории кода; доступ — только за аутентификацией.
- **R-PD3 (периметр обработки):** документировать, где обрабатываются данные (свой сервер vs внешний), что покидает контур ЦУ; парсинг через сторонний сервис — отдельное согласование.
- **R-PD4 (хранение/удаление):** план удаления данных после окончания курса.

## 8. Вне рамок v1 (явно НЕ делаем)

- **Любой студенческий и публичный интерфейс.** Продукт v1 — приватный инструмент только преподавателя. Сценарии Дамира (портфолио наружу) и Алии (приватный «что осталось») — НЕ v1 (возможные будущие отдельные продукты, не роли этого дашборда).
- **FR-5b** — семантический LLM-детект «разрыва смысла» (v2; v1 даёт только структурный прокси FR-5a).
- Оценка **качества** содержания артефакта ИИ.
- Доступ куратора/другого преподавателя, сводка по нескольким курсам/потокам, метрики retention, нотификации студентам.

## 9. Риски и снижение

| Риск | Влияние | Снижение |
|------|---------|----------|
| FR-5a даёт ложные разрывы (синонимы, переименования) | подрыв доверия к ядру на первой сработке | override (FR-10) + метрика точности (§3) + гейт живой валидации (§11) |
| Структурный прокси примут за «детект смысла» | завышенное обещание | честный критерий §4.2, FR-5b явно v2 |
| Именные данные студентов в приватном инструменте | вопросы СБ по обработке ПД | R-PD1 закрытый доступ + R-PD3/4 периметр и удаление |
| Преподаватель проваливается в чтение подряд | низкая адопция матрицы | вести сразу в разрывы (FR-5a > FR-4) |
