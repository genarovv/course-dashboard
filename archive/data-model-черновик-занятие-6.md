# Логическая модель данных — Course Dashboard

**Основание:** `product/prd.md` (v2.3) + словарь сущностей `product/dataobjects_draft2.md`. Модель логическая: СУБД и физические типы сознательно не выбираются (стек — занятие 7, ADR-002).

**Разделение канона** (по образцу PRD §8): атрибуты и обоснования сущностей канонически живут в словаре; связи, кратности и решения «состояние vs история» — здесь. Расхождение чинится в обоих файлах с пометкой даты.

Сущностей, которых нет в словаре, модель не добавляет; всё недостающее — в §5 «Вопросы».

---

## 1. ER-диаграмма (DBML)

Каждая связь снабжена обоснованием со ссылкой на требование. Связи, которые нечем обосновать, не нарисованы — они в §5.

```dbml
Enum git_host { GitLab GitHub }
Enum artifact_role { interview persona user_story prd data_model architecture plan code tests }
Enum availability { available unavailable }
Enum snapshot_status { found partial not_found }
Enum partial_reason { inexact_name wrong_place template_copy } // BR-3
Enum sync_trigger { schedule manual }                          // FR-8
Enum sync_status { in_progress completed partial failed }      // NFR-2, NFR-4
Enum rubric_type { edge step }                                 // FR-5 / FR-11
Enum verdict_value { ok break deferred }                       // FR-5: цела / разрыв / отложено (LLM недоступна)
Enum confidence_level { high medium low }                      // FR-5; на защите подсветка только high (риск §11)
Enum card_status { done deferred }                             // FR-11

Table system_user {
  id uuid [pk]
  username varchar
  password_hash varchar
  failed_attempts int
  locked_until timestamp [null]
  Note: 'FR-0/R-PD1. Связей нет: v1 однопользовательский (§3)'
}

Table git_credential {
  id uuid [pk]
  git_host git_host
  encrypted_token varchar
  is_valid bool
  checked_at timestamp
  Note: 'FR-3, NFR-3. Связь с repository — через домен git_host, FK избыточен (§5, В6)'
}

Table repository {
  id uuid [pk]
  repo_url varchar          // не ПДн — решение CEO 2026-07-02
  git_host git_host
  default_branch varchar
  added_at timestamp
  availability availability
  last_ok_sync_at timestamp [null]
}

Table lesson {
  id uuid [pk]
  number int                // упорядоченность нужна FR-7
  title varchar
  Note: 'Даты занятия в словаре нет — блокирует FR-7, см. §4 и §5 (В5)'
}

Table artifact_def {
  id uuid [pk]
  lesson_id uuid [ref: > lesson.id] // FR-2: чек-лист задаёт ожидаемые артефакты «по каждому занятию»
  role artifact_role
  expected_pattern varchar
  template_relative_path varchar [null] // FR-4/BR-3: детект заготовки
}

Table edge_def {
  id uuid [pk]
  source_role artifact_role // связь с artifact_def — через домен role, не FK (§5, В4)
  target_role artifact_role
  rubric_id uuid [ref: > rubric.id] // FR-2: «для каждого ребра — версионированная рубрика»; ссылка на действующую версию
}

Table rubric {
  id uuid [pk]              // одна строка = одна версия (append-only, §3)
  type rubric_type
  artifact_role artifact_role [null] // только type=step (FR-11)
  version varchar
  text text
  items json [null]         // только type=step: пункты чек-листа [{item_id, text}]
  created_at timestamp
}

Table sync_run {
  id uuid [pk]
  started_at timestamp
  completed_at timestamp [null]
  triggered_by sync_trigger
  status sync_status
}

Table artifact_snapshot {
  id uuid [pk]
  sync_run_id uuid [ref: > sync_run.id]         // FR-8/§13: наблюдение существует только как результат обхода («актуально на ЧЧ:ММ»)
  repository_id uuid [ref: > repository.id]     // FR-4: статус всегда о конкретном репозитории (строка матрицы)
  artifact_def_id uuid [ref: > artifact_def.id] // FR-4: статус имеет смысл только относительно ожидаемого артефакта (столбец матрицы)
  status snapshot_status
  partial_reason partial_reason [null]
  file_path varchar [null]
  source_commit_sha varchar [null]              // FR-9: устойчивость к force-push
  content_hash varchar [null]                   // FR-8 инкрементальность; «версия документа» в четвёрке FR-5 и тройке FR-11
  observed_at timestamp
}

Table coherence_verdict {
  id uuid [pk]
  edge_def_id uuid [ref: > edge_def.id]                    // FR-5: проверка определена ребром «для каждого ребра A→B»
  source_snapshot_id uuid [ref: > artifact_snapshot.id]    // FR-5: «версия документа A» в четвёрке привязки
  target_snapshot_id uuid [ref: > artifact_snapshot.id]    // FR-5: «версия документа B» в четвёрке привязки
  rubric_id uuid [ref: > rubric.id]                        // FR-5: «версия рубрики» в четвёрке; строка rubric = конкретная версия
  llm_model varchar                                        // FR-5: «модель» в четвёрке — значение, сущности LLM-модели в словаре нет и не нужно
  verdict verdict_value
  confidence confidence_level
  entities_checked int
  entities_found int
  entities_excluded int
  entities_lost int
  points json               // ≤5 точек: [{entity, source_quote, what_searched_in_target}]
  notes text [null]
  computed_at timestamp     // NFR-1: проверка асинхронна, завершается вне обхода — потому не FK на sync_run (§5, В1)
}

Table step_quality_card {
  id uuid [pk]
  snapshot_id uuid [ref: > artifact_snapshot.id] // FR-11: «версия документа» в тройке привязки
  rubric_id uuid [ref: > rubric.id]              // FR-11: «версия рубрики шага» в тройке
  llm_model varchar                              // FR-11: «модель» в тройке
  status card_status
  coverage json             // [{item_id, verdict: done|partial|missing}] — item_id из rubric.items
  remarks json              // ≤5 замечаний [{quote, comment}]
  notes text [null]
  computed_at timestamp     // как у coherence_verdict (NFR-1)
}

Table override {
  id uuid [pk]
  coherence_verdict_id uuid [null, ref: > coherence_verdict.id] // FR-10: отметка гасит конкретный ложный разрыв
  step_quality_card_id uuid [null, ref: > step_quality_card.id] // FR-11: «ложное замечание гасится отметкой (аналог FR-10)»
  reason varchar
  created_at timestamp
  revoked_at timestamp [null] // FR-10: отметка обратима; снятие — не удаление
  Note: 'XOR: заполнена ровно одна из двух ссылок'
}
```

**Правила целостности, не выразимые в DBML:**

- `override`: ровно одна из ссылок (`coherence_verdict_id` XOR `step_quality_card_id`) непуста.
- У `coherence_verdict` оба снапшота принадлежат одному `repository`, а их `artifact_def.role` совпадают с `source_role`/`target_role` ребра (иначе четвёрка бессмысленна).
- На одну четвёрку `(source content_hash, target content_hash, rubric_id, llm_model)` — не более одного вердикта (FR-5: «пока она не изменилась, вердикт не пересчитывается»); аналогично для тройки `step_quality_card`.
- На одну находку — не более одной **активной** (`revoked_at is null`) отметки; история отметок не ограничена.

## 2. Кратности: граничный анализ

Для каждой связи — два вопроса: «может ли A существовать без B?» (минимум) и «может ли у A быть несколько B?» (максимум), в обе стороны.

**R1. ArtifactDef → Lesson**
- ArtifactDef без Lesson? Нет — FR-2: чек-лист задаётся «по каждому занятию». Несколько занятий у одного ArtifactDef? Нет — определение привязано к одному занятию. → ровно 1.
- Lesson без ArtifactDef? Да — вводное занятие без сдачи допустимо. Несколько ArtifactDef у занятия? Да — ~18 артефактов на курс (§1). → 0..N.

**R2. EdgeDef → Rubric**
- EdgeDef без Rubric? Нет — FR-2: «для каждого ребра — версионированную рубрику»; ребро без рубрики непроверяемо (FR-5). Несколько действующих рубрик у ребра? Нет — действующая одна; прошлые версии остаются строками `rubric`, на которые ссылаются старые вердикты. → ровно 1.
- Rubric без EdgeDef? Да — устаревшая версия, на которую ребро уже не указывает. Несколько рёбер у одной рубрики? Переиспользование PRD не запрещает. → 0..N.

**R3. ArtifactSnapshot → SyncRun**
- Snapshot без SyncRun? Нет — наблюдение существует только как результат обхода (определение «обход» §0, FR-8). Несколько обходов у снапшота? Нет — снапшот фиксирует факт одного прочтения. → ровно 1.
- SyncRun без Snapshot? Да — обход упал до первого чтения (`failed`, NFR-2) или инкрементально не нашёл изменений (FR-8). Несколько снапшотов у обхода? Да — до 25 репо × артефакты. → 0..N.

**R4. ArtifactSnapshot → Repository**
- Snapshot без Repository? Нет — статус всегда о чьём-то репозитории (FR-4). → ровно 1.
- Repository без Snapshot? Да — опоздавший только что добавлен, обход ещё не шёл (FR-1). Несколько снапшотов у репозитория? Да — история наблюдений по обходам (§13). → 0..N.

**R5. ArtifactSnapshot → ArtifactDef**
- Snapshot без ArtifactDef? Нет — «есть/нет/частично» имеет смысл только относительно ожидаемого артефакта (FR-4: «сверяет с эталонным чек-листом»). → ровно 1.
- ArtifactDef без Snapshot? Да — только что добавлен в конфиг (рёбра и артефакты включаются по мере курса, §12). Несколько снапшотов у ArtifactDef? Да — по репозиториям и обходам. → 0..N.

**R6. CoherenceVerdict → EdgeDef**
- Verdict без EdgeDef? Нет — FR-5: проверка выполняется «для каждого ребра A→B». Несколько рёбер у вердикта? Нет — вердикт про одну пару ролей. → ровно 1.
- EdgeDef без Verdict? Да — ребро добавлено, проверка ещё не шла. Несколько вердиктов у ребра? Да — 25 репозиториев × версии пар. → 0..N.

**R7–R8. CoherenceVerdict → ArtifactSnapshot (source и target)**
- Verdict без снапшота-источника/приёмника? Нет — четвёрка FR-5 включает версии обоих документов; если документа нет в репозитории (`not_found`), проверка ребра не запускается — отсутствие файла ловит FR-4, а не FR-5. → ровно 1 + 1.
- Snapshot без вердиктов? Да — версия документа могла не участвовать в проверках (LLM недоступна — `deferred`/нет записи). Несколько вердиктов у снапшота? Да — документ участвует в нескольких рёбрах конвейера (§12) и пересчитывается при смене рубрики или модели (четвёрка меняется без смены документа). → 0..N.

**R9. CoherenceVerdict → Rubric**
- Verdict без Rubric? Нет — FR-5: проверка «по версионированной рубрике», версия входит в четвёрку. Несколько рубрик у вердикта? Нет — одна версия на проверку. → ровно 1.
- Rubric без Verdict? Да — свежесозданная версия. Несколько вердиктов у версии рубрики? Да. → 0..N.

**R10. StepQualityCard → ArtifactSnapshot**
- Card без Snapshot? Нет — FR-11: тройка привязки включает версию документа. → ровно 1.
- Snapshot без Card? Да — FR-11 это v2, и прогон мог не выполняться. Несколько карточек у снапшота? Да — пересчёт при смене версии рубрики шага или модели. → 0..N.

**R11. StepQualityCard → Rubric**
- Card без Rubric? Нет — FR-11: прогон = документ + рубрика шага. → ровно 1 (и только `type=step` — правило целостности).
- Rubric без Card? Да. Несколько карточек у версии рубрики? Да — все репозитории × версии документов. → 0..N.

**R12–R13. Override → CoherenceVerdict / StepQualityCard**
- Override без находки? Нет — FR-10: отметка гасит конкретную находку; ровно одна из двух ссылок (XOR). → 0..1 + 0..1 с XOR.
- Находка без Override? Да — большинство находок не оспорены. Несколько отметок у находки? Да — отметка обратима (FR-10): поставлена → снята (`revoked_at`) → поставлена вновь; активная максимум одна (правило целостности). → 0..N.

## 3. Состояние vs история — решение по каждой сущности

Сигналы PRD, требующие истории: **(а) динамика** — FR-7 «без новых артефактов ≥2 занятия подряд», FR-8 «перечитываются только изменения»; **(б) доказательство прошлого** — FR-9 «хронология артефактов с местами разрывов» на защите, FR-10 «отметка обратима», риск §11 «мигание вердикта»; **(в) внешняя сторона переписывает задним числом** — FR-9 `force-push`: git-история репозитория переписываема студентом. Плюс прямое указание §13: «модель данных — история наблюдений, не текущее состояние».

| Сущность | Решение | Обоснование (одна строка) |
|----------|---------|---------------------------|
| SystemUser | **состояние** | FR-0 нужен только текущий счётчик попыток и блокировка; требований к истории входов в PRD нет |
| GitCredential | **состояние** | NFR-3: секрет перезаписывается, лишних копий токена не храним |
| Repository | **состояние** | FR-6: слепая зона — про «сейчас» («при восстановлении доступа уходят автоматически»); динамику сдач хранит ArtifactSnapshot, не Repository |
| Lesson | **состояние** | справочник-конфиг, сигналов (а)/(б)/(в) нет |
| ArtifactDef | **состояние** | FR-2: правка чек-листа пересчитывает статусы вперёд; реконструкция прошлых конфигов не требуется (оговорка О4) |
| EdgeDef | **состояние** | то же, что ArtifactDef; рёбра включаются по мере курса (§12) |
| Rubric | **история (append-only версии)** | (б) FR-5: вердикт привязан к версии рубрики, golden set требует репродуцируемости → правка = новая строка, старые неизменяемы |
| SyncRun | **история** | (а) FR-8 «актуально на ЧЧ:ММ»: каждая строка — состоявшийся обход, перезаписывать нечего |
| ArtifactSnapshot | **история (append-only)** | все три сигнала: (а) FR-7 считается по последовательности наблюдений; (б) FR-9 хронология на защите; (в) force-push — наша запись с `source_commit_sha` остаётся, когда git-история уже переписана (§13 прямо) |
| CoherenceVerdict | **история (append-only)** | (б) FR-9 доказательная цепочка + риск §11: новая четвёрка = новая строка, старые вердикты не «мигают» |
| StepQualityCard | **история (append-only)** | FR-11 наследует паттерн ядра: привязка к тройке, пересчёт только новой строкой |
| Override | **история (append-only)** | (б) FR-10 обратимость: снятие = `revoked_at`, не удаление — на защите (риск §11 «не отменён преподавателем») видно и текущее решение, и его историю |

**Как модель отвечает на три типа сигналов:**

- **(а) FR-7 «хроники»:** «нет новых артефактов 2 занятия подряд» = в последовательности снапшотов репозитория нет строк с новым `content_hash` за два межзанятийных интервала. Вычислимо только по append-only истории — по перезаписываемому состоянию «последний статус» динамика невосстановима. (Ограничение: нужна дата занятия — см. О2/В5.)
- **(б) FR-9 защита:** хронологическая цепочка = снапшоты и вердикты репозитория, упорядоченные по `observed_at`/`computed_at`, каждый с цитатами (`points`) — «показать, как было» не требует пересчёта.
- **(в) force-push:** студент переписал git-историю — строки `artifact_snapshot` с `source_commit_sha` уже у нас и не зависят от текущего состояния репозитория; доказательство сохраняется даже для исчезнувших коммитов.

## 4. Что модель не выполняет или выполняет с оговорками

- **О1. FR-6, половина «не добавлены»:** модель видит только добавленные репозитории — студент, чьего репозитория нет в системе, невидим. «Не добавленные» вычислимы только при сверке с внешним CSV-реестром (BR-6) в момент импорта; постоянного представления у этой части слепой зоны нет.
- **О2. FR-7 «хроники» — не вычислим без даты занятия:** в словаре у Lesson нет даты, а «2+ занятия подряд» требует сопоставить обходы занятиям. Блокер уровня словаря → В5.
- **О3. FR-1 «сводка N/M/K»:** дубликаты отсеиваются при импорте и не хранятся; сводка — результат операции, не персистентные данные. История импортов отсутствует (сущности нет в словаре).
- **О4. FR-2 «пересчёт при изменении чек-листа»:** выполняется вперёд, но ArtifactDef/EdgeDef не версионируются — реконструировать «как выглядела матрица до правки конфига» нельзя. FR-9 не страдает: вердикты привязаны к версиям рубрик и документов, а не к конфигу.
- **О5. NFR-1 статус «проверяется»:** не хранится; отображается как вычисляемое состояние «есть актуальная четвёрка, вердикта нет, обход жив». Хранимое `deferred` покрывает только «LLM недоступна».
- **О6. FR-9 «карточка студента»:** выполняется как карточка репозитория (решение CEO 2026-07-02 — ФИО вне инструмента); поиск по фамилии внутри инструмента невозможен, сопоставление — на стороне преподавателя.
- **R-PD1..R-PD4 — выполняются:** доступ закрыт (SystemUser), токен шифруется (GitCredential), ПДн в данных нет (решение CEO: логин в URL не ПДн) → R-PD3/R-PD4 тривиальны.

## 5. Вопросы (связи и элементы, не вошедшие в модель)

- **В1. `sync_run_id` у вердиктов и карточек (есть в словаре draft 2)** — связь с обходом не обоснована требованиями: NFR-1 делает проверки асинхронными, они завершаются вне обхода; вердикт привязан к четвёрке, а не к обходу. В модели заменена атрибутом `computed_at`. Словарь синхронизирован 2026-07-02.
- **В2. `repository_id` у вердиктов и карточек (был в словаре)** — транзитивно выводится через снапшоты; прямой FK — денормализация с риском противоречия (снапшоты одного вердикта из разных репо). Убран; вместо него правило целостности «оба снапшота одного репозитория». Словарь синхронизирован.
- **В3. `rubric_version` строкой у вердикта/карточки (был в словаре)** — заменён на FK `rubric_id`: строка `rubric` и есть конкретная версия; требует неизменяемости строк (решено в §3: append-only). Словарь синхронизирован.
- **В4. EdgeDef ↔ ArtifactDef** — связаны через домен `role`, не FK. Если одна роль когда-нибудь встретится в двух занятиях (например, `code` на занятиях 8 и 9), потребуется справочник ролей — такой сущности в словаре нет. Пока роль уникальна в рамках курса — enum достаточно.
- **В5. У Lesson нет даты** — блокирует FR-7 (О2). Предложение: добавить `date` в словарь (атрибут, не сущность). Требует решения при синхронизации словаря.
- **В6. GitCredential ↔ Repository** — логическая связь «репозиторий обслуживается токеном своего хостинга» существует через домен `git_host` (2 значения); FK между таблицами избыточен.
