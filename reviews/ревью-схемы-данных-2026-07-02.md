**Объект:** словарь сущностей (draft 2) + логическая модель (шаг 2) — оба поглощены итоговым `../data-model.md`; страничные ссылки ниже (§, В-номера) — на черновики в `../archive/`.

## S1 (негативный: источник недоступен). Протух токен GitLab посреди планового обхода

Группа: 12 репо GitLab + 13 GitHub. Плановый обход 09:00, GitLab API → 401.

| Шаг | Таблица → строка |
|---|---|
| 1 | `sync_run (run-42, started_at=09:00, triggered_by=schedule, status=in_progress)` |
| 2 | `git_credential` (GitLab): `is_valid=false, checked_at=09:01` — FR-3 «обнови токен» ✓ |
| 3 | 13 репо GitHub: снапшоты пишутся штатно ✓ |
| 4 | 12 репо GitLab: снапшотов нет. `repository.availability` — ? **Развилка без ответа** |
| 5 | `sync_run: status=partial, completed_at=09:02` ✓ |

**Дыра 1:** ставить `unavailable` — FR-6 ложно сваливает 12 студентов в слепую зону (недоступен *наш токен*, не их репо); не ставить — в данных нет следа, что репо не проверялись. Enum не различает причину.
**Дыра 2:** `status=partial` не говорит, какие репозитории не покрыты — связи «обход × репозиторий с исходом» нет.
**Критичность: серьёзно** → С5.

## S2 (негативный: пользователь сделал не то). Повторный CSV: вариант URL + опечатка

| Шаг | Таблица → строка |
|---|---|
| 1 | В базе: `repository (rep-07, repo_url='…/course-work')` + история снапшотов |
| 2 | Импорт `…/course-work.git`: байтового совпадения нет → дубликат не распознан → `repository (rep-26)` |
| 3 | Матрица 26 строк; история студента расщеплена между rep-07 и rep-26 — FR-1 «без потери накопленной истории» подорван |

**Критичность: серьёзно** → С2, И6.

## S4 (негативный: данные задним числом). Force-push с подделкой дат коммитов

| Шаг | Таблица → строка |
|---|---|
| 1 | До: `artifact_snapshot (snap-100, sha=abc123, hash=h1, observed_at=06-20)`; `coherence_verdict (verd-9, break, high, points=[…])` |
| 2 | Force-push: abc123 стёрт, даты коммитов подделаны старыми |
| 3 | Обход: `artifact_snapshot (snap-200, sha=def456, hash=h2, observed_at=сегодня)`; старая строка нетронута ✓ |
| 4 | Хронология FR-9 — по `observed_at` наших наблюдений; подделка git-дат бессильна ✓ |

**Схема выдерживает ядро сценария.** Дыра в доказательности: контент нигде не хранится (только хеш и цитаты в `points`) — студент говорит «такого в файле не было», проверить цитату против стёртого коммита не по чему; `content_hash` доказывает «файл был другим», но не «в нём была эта фраза».
**Критичность: серьёзно** → С4.

## S5 (P0, сложнейшая логика: FR-5 + FR-10). Реверт документа — двойная идентичность вердикта

| Шаг | Таблица → строка |
|---|---|
| 1 | `snap-101 (B, hash=hB1)`; `verd-9 (source=snap-100, target=snap-101, rubric=rub-3, model=m1, break, high)`; `override (ov-1, → verd-9, revoked_at=null)` |
| 2 | Студент правит B: `snap-150 (hash=hB2)` → `verd-15 (break)`; ov-1 не действует — по FR-10 корректно ✓ |
| 3 | Студент откатывает правку: `snap-180 (hash=hB1)` — хеш совпал со старым |
| 4 | Четвёрка `(hA,hB1,rub-3,m1)` занята verd-9 → «не пересчитывается». Но verd-9 → snap-101, не snap-180. **Разложить нельзя** |

**Противоречие (блокер):** идентичность вердикта задана и FK на `snapshot_id`, и четвёркой `content_hash` — при реверте они расходятся: по FK вердикта нет, вставить новый — нарушить уникальность четвёрки. Дашборд для snap-180 либо молчит, либо дублирует. Повисает и FR-10: действует ли ov-1 на «возрождённую» четвёрку — схема не отвечает. Реверт — не экзотика.
**Критичность: БЛОКЕР** → Б1.

## S6 (P0: FR-8 + FR-7). Инкрементальность съедает «хроники» и «актуально на»

| Шаг | Раскладка |
|---|---|
| 1 | rep-07 две недели без изменений; `sync_run` ×14 ✓ |
| 2 | FR-8 «перечитываются только изменения» → снапшоты не пишутся; новых строк по rep-07 ноль |

| 4 | FR-8 «актуально на ЧЧ:ММ»: последний снапшот двухнедельной давности при сегодняшнем подтверждении «без изменений» — матрица **врёт** |

**Дыры (блокер):** (1) нет представления факта «проверено, изменений нет» (нужна пообходная запись охвата `sync_run × repository`); (2) у `lesson` нет даты (признано черновиком, В5) — «2+ занятия подряд» не к чему привязать.
**Критичность: БЛОКЕР** → Б2.

## S7 (P0: FR-9 + риск §11). Защита: подсветка только high и не отменённых

| Шаг | Раскладка |
|---|---|
| 1 | Карточка rep-07: снапшоты + вердикты по `observed_at`/`computed_at` ✓ |
| 2 | `verd-9 (break, high)` + активная ov-1 → не подсвечивать: `override where coherence_verdict_id=verd-9 and revoked_at is null` ✓ — по логической модели |
| 3 | Та же раскладка по словарю draft 2: `override (finding_type=coherence, finding_id=verd-9)` — **другая структура таблицы** |

**Противоречие (раскладывается двумя способами — между двумя канонами):** модель задаёт `override` как два nullable-FK с XOR, словарь — как полиморфную пару без FK; оба датированы одной синхронизацией. Полиморфный вариант вдобавок лишает ссылку проверяемости.
**Критичность: серьёзно. Статус: закрыто при сборке итогового `data-model.md`** — принят вариант «2 FK + XOR» (§1.12).

## S8 (P0: FR-5). LLM недоступна → `deferred` → LLM вернулась

| Шаг | Раскладка |
|---|---|
| 1 | LLM down при проверке пары → `coherence_verdict (verd-30, verdict=deferred, llm_model=?, confidence=?, entities_*=?)` — поля обязательные, модель не вызывалась: осмысленных значений **нет** |
| 2 | LLM вернулась → нужен настоящий вердикт по той же четвёрке: она занята deferred-строкой (уникальность И3), перезаписать нельзя (append-only И5) |

**Противоречие:** append-only × уникальность четвёрки × `deferred` — попарно совместимы, втроём нет. «Отложено» — состояние очереди, не вердикт. Смежно: статус «проверяется» тоже нигде не хранится (О5/В13).
**Критичность: серьёзно** → С1.
